# Architecture

**Analysis Date:** 2026-03-08

## Pattern Overview

**Overall:** Obsidian Plugin - Event-driven, lifecycle-managed extension

**Key Characteristics:**
- Plugin class extends Obsidian's `Plugin` base class, which provides the entire lifecycle API
- All plugin behavior is registered during `onload()` and automatically cleaned up on `onunload()`
- Settings are persisted via Obsidian's built-in `loadData()`/`saveData()` mechanism (stores to `data.json` in the plugin folder)
- UI is built using Obsidian's provided component classes (`Modal`, `PluginSettingTab`, `Notice`) rather than raw DOM manipulation

## Layers

**Plugin Entry Point:**
- Purpose: Lifecycle management - registers all commands, UI, listeners, and intervals
- Location: `src/main.ts`
- Contains: The default-exported `MyPlugin` class extending `Plugin`; the `SampleModal` class
- Depends on: `src/settings.ts`, the `obsidian` package
- Used by: Obsidian runtime (loads and unloads this class automatically)

**Settings Layer:**
- Purpose: Defines the shape of user-configurable data, provides defaults, and renders the settings UI
- Location: `src/settings.ts`
- Contains: `MyPluginSettings` interface, `DEFAULT_SETTINGS` constant, `SampleSettingTab` class
- Depends on: `obsidian` package, `MyPlugin` (reference back for `saveSettings`)
- Used by: `src/main.ts`

**Build Output:**
- Purpose: The single bundled file loaded by Obsidian at runtime
- Location: `main.js` (plugin root, generated - not committed)
- Generated by: esbuild from `src/main.ts` as entry point

## Data Flow

**Settings Persistence:**

1. `onload()` calls `this.loadSettings()` which merges stored `data.json` with `DEFAULT_SETTINGS` via `Object.assign`
2. User changes a value in `SampleSettingTab.display()` via the `onChange` handler
3. Handler mutates `this.plugin.settings` and calls `this.plugin.saveSettings()`
4. `saveSettings()` calls `this.saveData(this.settings)` which writes to `data.json`

**Command Execution:**

1. User triggers a command from the Command Palette or a keyboard shortcut
2. Obsidian invokes the registered `callback` or `editorCallback` from `this.addCommand(...)`
3. Callback creates UI components (e.g., `new SampleModal(this.app).open()`) or performs editor operations

**State Management:**
- All plugin state lives on the `MyPlugin` instance as `this.settings`
- No external state management library; Obsidian's `App` object provides access to vault, workspace, and file system

## Key Abstractions

**Plugin Class (`MyPlugin`):**
- Purpose: Root object - owns settings, registers all capabilities with Obsidian
- Examples: `src/main.ts`
- Pattern: Extends `Plugin`; uses `this.register*` helpers so all listeners/intervals are auto-cleaned on unload

**Settings Interface (`MyPluginSettings`):**
- Purpose: TypeScript contract for the shape of persisted user preferences
- Examples: `src/settings.ts`
- Pattern: Plain interface + `DEFAULT_SETTINGS` constant + `SampleSettingTab` class (all co-located)

**Modal (`SampleModal`):**
- Purpose: Transient UI overlay triggered by commands
- Examples: `src/main.ts` (currently defined inline)
- Pattern: Extends `Modal`; implements `onOpen()` and `onClose()` to build/tear down DOM content

## Entry Points

**Plugin Load:**
- Location: `src/main.ts` - `MyPlugin.onload()`
- Triggers: Obsidian loads the plugin (on app start or manual enable in settings)
- Responsibilities: Load settings, register ribbon icon, status bar item, commands, settings tab, DOM event listeners, intervals

**Build Entry:**
- Location: `src/main.ts` (consumed by `esbuild.config.mjs`)
- Triggers: `npm run dev` (watch) or `npm run build` (production)
- Responsibilities: Compiles and bundles all source into `main.js` at the plugin root

## Error Handling

**Strategy:** Minimal - no explicit error boundaries in the current scaffold

**Patterns:**
- `async/await` used for settings load/save; errors are not currently caught
- `Object.assign({}, DEFAULT_SETTINGS, await this.loadData())` provides a safe fallback if stored data is missing or partial

## Cross-Cutting Concerns

**Logging:** `console.log` (used in the registered interval example)
**Validation:** None - settings values are accepted as-is from the UI text input
**Authentication:** Not applicable - plugin operates fully locally within the user's vault
